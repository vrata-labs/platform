# Фаза M0.5 — приемка базового многопользовательского присутствия

## 1. Назначение документа

Этот документ является рабочим заданием для агента разработки.

Цель фазы — **закрыть базовое многопользовательское присутствие** в пространственной веб-комнате.

Под базовым присутствием понимается не красивый человекоподобный аватар, а надежный технический минимум:

- участник входит в комнату по ссылке;
- остальные участники видят его появление;
- остальные участники видят его перемещение;
- остальные участники понимают, куда он примерно повернут;
- голос включается только по явному действию пользователя;
- состояние микрофона передается в состояние комнаты;
- источник пространственного звука получает координаты из состояния участника;
- выход и повторный вход не оставляют остаточных участников;
- основное поведение подтверждено автоматическими проверками;
- финальная ручная проверка проводится только в конце и только для того, что невозможно надежно проверить автоматически.

Фаза закрывает надежность текущей нулевой версии. Она не расширяет продукт в сторону новых шаблонов, панели управления, человекоподобных аватаров или редактора сцены.

---

## 2. Контекст проекта для агента

Проект строит пространственные веб-комнаты для встреч, демонстраций и совместной работы.

Ключевая рамка:

- обычный браузер и мобильный браузер являются полноценными способами входа;
- виртуальная реальность является улучшением, а не отдельным продуктом;
- одна и та же ссылка на комнату должна работать для обычного браузера, мобильного браузера и устройства виртуальной реальности;
- состояние комнаты и голосовая связь должны оставаться разными слоями системы;
- состояние участников не должно храниться в голосовом слое;
- голосовая связь не должна заменять слой состояния комнаты.

В этой фазе агент работает только над надежностью базового присутствия.

Нельзя начинать:

- человекоподобные аватары;
- руки, пальцы, скелет, мимику, одежду;
- новые шаблоны комнат;
- полноценную панель управления;
- отдельную систему развертывания для клиентов;
- переписывание голосовой связи;
- замену текущего трехмерного движка;
- отдельный путь исполнения только для виртуальной реальности.

## 3. Главная цель фазы

Фаза считается закрытой, если система автоматически подтверждает следующие свойства:

1. Четыре браузерных участника могут одновременно находиться в одной комнате.
2. Каждый участник видит всех остальных участников.
3. Выход участника удаляет его визуальное и звуковое представление у остальных.
4. Повторный вход не создает остаточных участников.
5. В состоянии участника передается не только положение, но и ориентация.
6. Ориентация используется для отображения направления головы и корпуса удаленного участника.
7. Движение удаленного участника сглаживается и не выглядит как грубая телепортация по диагностическим метрикам.
8. Голосовой вход, выключение микрофона и отказ микрофона не ломают комнату.
9. Пространственный звук получает положение источника из состояния головы, тела или корня участника.
10. Режим виртуальной реальности проверяется автоматически через имитацию, а затем вручную на реальном устройстве.

Ручная проверка проводится только после прохождения автоматических проверок.

## 4. Термины

### Участник

Пользователь, подключенный к комнате.

В коде это запись в состоянии комнаты, а также локальное или удаленное отображение в клиентской части.

### Локальный участник

Участник, который управляется текущей страницей браузера.

### Удаленный участник

Участник, которого текущая страница получает из состояния комнаты и отображает в сцене.

### Корень участника

Базовая точка участника на полу.

Для обычного браузера это положение управляемого персонажа. Для устройства виртуальной реальности это условная точка, относительно которой считается положение головы.

### Голова участника

Положение камеры или шлема.

Для обычного браузера голова может быть вычислена из положения камеры. Для устройства виртуальной реальности голова должна соответствовать положению шлема.

### Поза

Положение вместе с ориентацией.

Положение:

- `x` — координата по горизонтальной оси;
- `y` — высота;
- `z` — координата по второй горизонтальной оси.

Ориентация:

- `yaw` — поворот влево-вправо вокруг вертикальной оси;
- `pitch` — наклон вверх-вниз;
- `roll` — наклон головы вбок. Для этой фазы необязателен.

### Сглаживание движения

Механизм, при котором удаленный участник не переставляется мгновенно из одной точки в другую, а плавно проходит между полученными состояниями.

### Остаточный участник

Ошибочное удаленное представление пользователя, который уже вышел из комнаты, но его тело, голова, звук или запись в диагностических данных остались активными у других клиентов.

### Диагностические данные

Структура данных, доступная из браузера через `window.__VRATA_DEBUG__`.

Она нужна для автоматических проверок. Проверки должны читать состояние комнаты программно, а не полагаться на ручной осмотр экрана.

## 5. Общие правила выполнения

Агент обязан соблюдать следующий порядок:

1. Сначала зафиксировать исходное состояние проекта.
2. Затем добавить или усилить автоматическую проверку.
3. Только после этого менять код.
4. После каждого изменения запускать целевую проверку.
5. После завершения рабочего пакета запускать связанные проверки.
6. В конце запустить единый набор проверок фазы.
7. После этого выполнить финальную ручную проверку.

Главное правило:

> Если поведение можно проверить автоматически, оно должно быть проверено автоматически.

Ручная проверка допустима только для:

- настоящего устройства виртуальной реальности;
- субъективной слышимости голоса;
- субъективного ощущения направления пространственного звука;
- десятиминутной живой демонстрации с реальными участниками.

## 9. Рабочий пакет 3 — расширение модели позы участника

### Цель

Состояние комнаты должно передавать не только координаты, но и ориентацию участника.

### Изменения в типах

Расширить текущий тип положения.

Минимально допустимый вариант:

```ts
export interface TransformState {
  x: number;
  y: number;
  z: number;
  yaw?: number;
  pitch?: number;
  roll?: number;
}
```

Для корня участника использовать:

```ts
rootTransform: {
  x: number;
  y: number;
  z: number;
  yaw: number;
}
```

Для головы участника использовать:

```ts
headTransform: {
  x: number;
  y: number;
  z: number;
  yaw: number;
  pitch: number;
}
```

Если `yaw`, `pitch` или `roll` отсутствуют в старом сообщении, клиент обязан считать их равными `0`.

### Последовательность обновлений

В состояние участника добавить:

```ts
seq: number;
clientTimeMs: number;
serverTimeMs?: number;
```

Пояснение:

- `seq` — монотонно растущий номер обновления от клиента;
- `clientTimeMs` — время на стороне клиента;
- `serverTimeMs` — время на стороне службы состояния комнаты, если доступно.

Это нужно, чтобы проверки и сглаживание могли отличить новое состояние от старого.

### Задачи

1. Обновить общие типы состояния.
2. Обновить службу состояния комнаты.
3. Обновить отправку локального состояния из клиентской части.
4. Обновить прием состояния удаленных участников.
5. Обновить отрисовку удаленного участника.
6. Обновить диагностические данные.
7. Сохранить обратную совместимость со старым форматом `x/y/z`.

### Автоматические проверки

Создать или обновить модульные проверки:

```text
apps/room-state/src/schema.test.ts
apps/runtime-web/src/pose.test.ts
```

Проверить:

- старый формат `x/y/z` преобразуется в позу с `yaw = 0`;
- новый формат с `yaw/pitch` сохраняется без потери;
- `seq` увеличивается при отправке обновлений;
- устаревшее обновление с меньшим `seq` не перетирает более новое состояние.

### Критерий готовности

- типы собираются без ошибок;
- старые сценарии комнаты продолжают работать;
- новые поля видны в `window.__VRATA_DEBUG__`;
- автоматическая проверка подтверждает, что ориентация передается.

## 10. Рабочий пакет 4 — отображение направления участника

### Цель

Удаленный участник должен показывать не только положение, но и направление.

### Визуальное поведение

К существующему телу и голове добавить простой указатель направления.

Допустимые варианты:

- короткий луч перед головой;
- треугольный указатель на полу;
- тонкая линия направления взгляда;
- небольшой маркер на голове.

Не делать сложный аватар.

### Задачи

1. Создать визуальный элемент направления.
2. Привязать тело к `rootTransform.yaw`.
3. Привязать голову к `headTransform.yaw` и `headTransform.pitch`.
4. Убедиться, что поворот не вычисляется только через `lookAt` между телом и головой.
5. Добавить в диагностические данные фактически примененный поворот.

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/presence-orientation.spec.ts
```

Сценарий:

1. Открыть наблюдателя: `/rooms/demo-room?debug=1`.
2. Открыть второго участника: `/rooms/demo-room?debug=1&bot=turn`.
3. Дождаться, что наблюдатель видит одного удаленного участника.
4. Считать из `remoteParticipants[0].head.yaw` несколько значений.
5. Проверить, что значения меняются.
6. Проверить, что фактически примененный поворот в диагностике тоже меняется.

### Критерий готовности

- удаленный участник имеет видимый указатель направления;
- автоматическая проверка подтверждает изменение ориентации;
- ориентация головы и корпуса не подменяется только координатами.

## 11. Рабочий пакет 5 — роботы-участники для автоматических проверок

### Цель

Автоматические проверки должны имитировать участников без ручного управления мышью, клавиатурой и устройством виртуальной реальности.

### Робот-участник

Робот-участник — это обычная страница комнаты, которая по параметру в адресе сама выполняет предсказуемые движения.

Адреса:

```text
/rooms/demo-room?debug=1&bot=idle
/rooms/demo-room?debug=1&bot=line
/rooms/demo-room?debug=1&bot=turn
/rooms/demo-room?debug=1&bot=square
```

### Режимы

#### `bot=idle`

Участник стоит на месте.

Нужен для проверки входа, выхода и количества участников.

#### `bot=line`

Участник движется вперед-назад по линии.

Нужен для проверки передачи координат.

#### `bot=turn`

Участник стоит на месте, но меняет `yaw`.

Нужен для проверки ориентации.

#### `bot=square`

Участник идет по квадрату и поворачивается на углах.

Нужен для проверки движения, поворота и сглаживания.

### Дополнительные параметры

Начальная позиция:

```text
botStart=x,z
```

Пример:

```text
/rooms/demo-room?debug=1&bot=line&botStart=2,0
```

Скорость:

```text
botSpeed=0.8
```

### Задачи

1. Проверить, есть ли уже режим `bot=line`.
2. Добавить недостающие режимы.
3. Сделать движение детерминированным: одинаковые параметры дают одинаковое движение.
4. Роботы не должны требовать реального микрофона.
5. Роботы не должны требовать устройства виртуальной реальности.

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/bot-participants.spec.ts
```

Проверить:

- `bot=idle` подключается и появляется в присутствии;
- `bot=line` меняет координаты;
- `bot=turn` меняет `yaw`;
- `bot=square` меняет координаты и `yaw`.

### Критерий готовности

- все режимы робота работают без ручного ввода;
- проверки могут создавать четыре участника с разными сценариями;
- роботы не требуют микрофона;
- роботы не требуют устройства виртуальной реальности.

## 12. Рабочий пакет 6 — проверка четырех участников

### Цель

Автоматически подтвердить главный критерий M0: от двух до четырех участников в одной комнате.

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/four-participants-presence.spec.ts
```

Сценарий:

1. Открыть четыре страницы браузера.
2. Каждой странице дать уникальные параметры:

```text
?debug=1&bot=idle&name=BotA
?debug=1&bot=line&name=BotB&botStart=2,0
?debug=1&bot=turn&name=BotC&botStart=-2,0
?debug=1&bot=square&name=BotD&botStart=0,2
```

3. Дождаться подключения всех участников.
4. На каждой странице проверить:

```ts
remoteAvatarCount === 3
```

5. Через служебный адрес присутствия, если он есть, проверить:

```ts
presence.items.length >= 4
```

6. Проверить, что у каждого участника уникальный `participantId`.

## 14. Рабочий пакет 8 — сглаживание движения и пределы скачков

### Цель

Автоматически подтвердить, что движение удаленных участников не является грубой телепортацией.

### Метрики

В диагностические данные каждого удаленного участника добавить:

```ts
updateHz: number;
staleMs: number;
maxObservedJumpM: number;
interpolationDelayMs: number;
```

Пояснение:

- `updateHz` — примерная частота обновлений в секунду;
- `staleMs` — сколько миллисекунд прошло с последнего обновления;
- `maxObservedJumpM` — максимальное замеченное смещение между соседними примененными состояниями;
- `interpolationDelayMs` — задержка сглаживания движения.

Мягкие пороги для этой фазы:

```text
updateHz >= 3
staleMs <= 1500
maxObservedJumpM <= 1.25
```

Первое появление участника не должно считаться скачком.

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/motion-smoothing.spec.ts
```

Сценарий:

1. Открыть наблюдателя.
2. Открыть `bot=line`.
3. Подождать 20 секунд.
4. Проверить у наблюдателя:

```ts
remoteParticipants[0].updateHz >= 3
remoteParticipants[0].staleMs < 1500
remoteParticipants[0].maxObservedJumpM <= 1.25
```

5. Повторить с `bot=square`.

### Критерий готовности

- проверка проходит для линейного движения;
- проверка проходит для движения по квадрату;
- начальное появление участника не портит статистику;
- явные телепортации ловятся автоматической проверкой.

## 15. Рабочий пакет 9 — голосовой вход и состояние микрофона

### Цель

Голос должен быть проверяемым автоматически настолько, насколько это возможно без ручного прослушивания.

### Что проверяется автоматически

Автоматическая проверка должна подтверждать:

- микрофон не запрашивается при загрузке страницы;
- микрофон запрашивается только после явного действия пользователя;
- успешное подключение меняет состояние голосового слоя;
- публикация звука начинается;
- выключение микрофона меняет состояние локального участника;
- выключение микрофона видно другим участникам;
- отказ микрофона не ломает комнату.

### Переходы состояния

Минимальные переходы:

```text
not_joined -> joining -> joined
joined -> muted
muted -> joined
joined -> not_joined
joining -> failed
joined -> degraded
```

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/voice-state.spec.ts
```

#### Сценарий A — успешное подключение

1. Открыть страницу A с разрешенным искусственным микрофоном.
2. Нажать кнопку включения звука.
3. Дождаться:

```ts
media.audioState === "joined"
media.publishedAudio === true
```

4. Открыть страницу B.
5. Проверить на B, что удаленный участник A имеет:

```ts
activeAudio === true
muted === false
```

#### Сценарий B — выключение микрофона

1. На A нажать выключение микрофона.
2. Проверить на A:

```ts
media.muted === true
```

3. Проверить на B:

```ts
remoteParticipants[A].muted === true
```

#### Сценарий C — отказ микрофона

1. Открыть страницу с запретом микрофона.
2. Нажать кнопку включения звука.
3. Проверить:

```ts
media.audioState === "degraded" || media.audioState === "failed"
```

4. Проверить, что комната остается подключенной:

```ts
roomStateConnected === true || roomStateMode === "api_fallback"
```

### Критерий готовности

- автоматические проверки подтверждают переходы состояния звука;
- отказ микрофона не ломает присутствие;
- выключение микрофона видно другому участнику;
- голосовой слой не становится источником состояния комнаты.

## 16. Рабочий пакет 10 — пространственный звук через диагностику

### Цель

Проверить не субъективное качество звучания, а правильную техническую привязку звука к положению участника.

### Что проверяется автоматически

1. Для удаленного участника создается звуковой узел.
2. Источник звука привязан к голове, телу или корню участника.
3. При движении удаленного участника положение источника звука меняется.
4. При движении локального участника положение слушателя меняется.
5. Если пространственный звук отключен, обычный голос остается резервным режимом.

### Изменения

В диагностические данные добавить:

```ts
spatialAudio: {
  enabled: boolean,
  fallback: boolean,
  listener: {
    x: number,
    y: number,
    z: number,
    yaw: number
  },
  remoteSources: [
    {
      participantId: string,
      x: number,
      y: number,
      z: number,
      attachedTo: "head" | "body" | "root"
    }
  ]
}
```

Добавить параметр отключения пространственного звука:

```text
?spatial=0
```

### Автоматическая проверка

Создать файл:

```text
tests/e2e/m0.5/spatial-audio-diagnostics.spec.ts
```

Сценарий:

1. Открыть A как наблюдателя.
2. Открыть B как `bot=line`.
3. Если возможно, подключить звук через искусственный микрофон.
4. Дождаться появления удаленного источника звука у A.
5. Проверить:

```ts
spatialAudio.enabled === true
spatialAudio.remoteSources.length >= 1
```

6. Проверить, что `attachedTo` имеет одно из значений:

```ts
"head" | "body" | "root"
```

7. Через несколько секунд проверить, что координаты источника изменились вслед за B.
8. Повторить запуск с `?spatial=0` и проверить, что голосовой слой не падает.

### Критерий готовности

- пространственный звук имеет проверяемые диагностические координаты;
- координаты источника следуют за участником;
- выключение пространственного звука не ломает обычный голос;
- финальная оценка на слух остается только в ручной приемке.

### Критерий готовности

## 21. Рабочий пакет 15 — финальная ручная приемка

### Цель

Проверить только то, что невозможно надежно подтвердить автоматическими проверками.

### Состав участников

Минимум:

- обычный браузер на компьютере;
- второй обычный браузер или телефон;
- устройство виртуальной реальности;
- при возможности четвертый участник в обычном браузере.

### Сценарий

1. Открыть одну и ту же комнату по ссылке.
2. Убедиться, что все участники видят всех.
3. Участник в виртуальной реальности двигается и поворачивает голову.
4. Обычный браузер видит движение и поворот головы.
5. Обычный браузер двигается.
6. Участник в виртуальной реальности видит это движение.
7. Все участники включают голос.
8. Один участник выключает микрофон.
9. Остальные видят изменение состояния микрофона.
10. Один участник выходит.
11. Остальные видят исчезновение участника.
12. Участник входит повторно.
13. Остальные видят одного нового участника, без остаточного участника.
14. Проверить пространственный звук на слух: направление и расстояние должны быть хотя бы примерно понятны.
15. Продержать комнату открытой не менее 10 минут.

### Критерий успешной ручной приемки

- нет зависаний комнаты;
- нет остаточных участников после выхода;
- движение участника в виртуальной реальности видно в обычном браузере;
- движение обычного участника видно в виртуальной реальности;
- голос слышен;
- выключение микрофона понятно;
- пространственный звук не ломает обычную слышимость;
- за 10 минут комната не деградирует до неработоспособного состояния.

## 26. Краткая инструкция для агента

Закрой фазу `M0.5 — приемка базового присутствия`.

Не добавляй новые продуктовые возможности. Не делай человекоподобных аватаров. Не переписывай архитектуру.

Сначала сделай состояние комнаты измеримым через диагностические данные. Затем добавь ориентацию участника, роботов для автоматических проверок, проверки четырех участников, входа/выхода/повторного входа, сглаживания движения, состояния голоса, пространственного звука и имитации режима виртуальной реальности.

Фаза закрыта только после прохождения `pnpm run test:m0.5`, всех обычных проверок проекта и одной финальной ручной проверки с реальным устройством виртуальной реальности.
