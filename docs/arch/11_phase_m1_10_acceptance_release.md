# M1.10 — приемка и выпуск фазы

## Цель

Формально закрыть M1-MEDIA как готовую продуктовую фазу.

Фаза закрыта не тогда, когда отдельные объекты «вроде работают», а когда весь сценарий деловой комнаты подтвержден автоматическими проверками и одной финальной ручной проверкой.

## Итоговый сценарий

1. Ведущий авторизуется и входит в комнату.
2. Два или три участника входят как гости или участники.
3. В комнате есть несколько медийных поверхностей.
4. Ведущий выбирает главную поверхность.
5. Ведущий включает трансляцию экрана.
6. Участники видят поток на главной поверхности.
7. Ведущий выбирает другую поверхность.
8. Ведущий включает доску.
9. Участник с правами рисует на доске.
10. Ведущий в виртуальной реальности выбирает поверхность лучом.
11. Ввод лучом попадает в протокол поверхности.
12. Остановка трансляции не очищает доску.
13. Очистка доски не останавливает трансляцию.
14. Выход ведущего переводит активные объекты в безопасное состояние.

## Единая команда проверки

Добавить команду:

```json
{
  "scripts": {
    "test:m1-media": "pnpm run lint && pnpm run typecheck && pnpm run test && pnpm exec playwright test tests/e2e/m1-media"
  }
}
```

Если текущая структура команд отличается, агент обязан использовать существующие команды проекта и добавить ближайший эквивалент.

## Автоматическая приемочная проверка

Создать:

```text
tests/e2e/m1-media/m1-media-acceptance.spec.ts
```

Сценарий:

1. Открыть ведущего:

```text
/rooms/demo-room?debug=1&role=host
```

2. Открыть участника:

```text
/rooms/demo-room?debug=1&role=member
```

3. Открыть гостя:

```text
/rooms/demo-room?debug=1&role=guest
```

4. Убедиться, что все видят друг друга.
5. Убедиться, что есть минимум две поверхности.
6. Ведущий создает объект трансляции экрана на `main-screen` через подмененный медиа-провайдер.
7. Участник видит активную трансляцию.
8. Ведущий создает доску на `whiteboard-wall`.
9. Участник рисует штрих.
10. Гость видит доску, но не может рисовать, если политика это запрещает.
11. Имитация VR-луча выбирает поверхность и отправляет событие ввода.
12. Остановка трансляции не очищает доску.
13. Очистка доски не останавливает трансляцию.
14. Все отладочные данные согласованы.

## Финальная ручная проверка

Ручная проверка нужна только для того, что нельзя надежно проверить автоматически:

- реальный выбор окна, экрана или вкладки;
- реальное устройство виртуальной реальности;
- удобство попадания лучом в поверхность;
- субъективное качество видеопотока на поверхности.

### Состав проверки

Минимум:

- ведущий на обычном компьютере;
- участник на обычном компьютере или телефоне;
- участник или ведущий в устройстве виртуальной реальности.

### Ручной сценарий

1. Ведущий входит с ролью `host`.
2. Участники входят по ссылке.
3. Ведущий запускает реальную трансляцию вкладки или окна.
4. Участники видят поток на поверхности.
5. Ведущий включает доску на другой поверхности.
6. Участник рисует на доске.
7. Устройство виртуальной реальности выбирает поверхность лучом.
8. Устройство виртуальной реальности отправляет клик или рисование на доске.
9. Ведущий останавливает трансляцию.
10. Доска остается активной.
11. Ведущий очищает доску.
12. Участники видят очистку.
13. Один участник выходит и входит повторно.
14. Состояние поверхностей остается корректным.

## Критерии закрытия фазы

Фаза закрыта, если:

- `pnpm run test:m1-media` проходит;
- обычные проверки проекта проходят;
- финальная ручная проверка выполнена;
- документы обновлены;
- ограничения явно записаны;
- новые возможности реализованы как медийные объекты или расширения, а не как правки ядра под каждый случай.

## Что не считать закрытием

- Трансляция экрана работает только в отдельном HTML-блоке, а не на медийной поверхности.
- Доска работает только локально.
- Ввод из VR не проходит через общий протокол.
- Несколько поверхностей существуют визуально, но состояние у них общее.
- Роль ведущего не проверяется на серверной стороне.
- Новые объекты требуют правки ядра.
- Реальный захват экрана не проверен вручную.
