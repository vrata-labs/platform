# План: экспорт и подключение сцен SenseTower в noah (v1)

## Цель

Подключить в `noah` 1-2 реальные сцены из SenseTower через воспроизводимый pipeline: экспорт из Unity-источника в веб-формат, описание сцены в простом универсальном manifest-формате и выбор разных сцен для разных комнат без правок runtime под каждую сцену.

## Не-цель

- Не читать `.unity` сцены напрямую в браузере или в `apps/runtime-web`.
- Не переносить Unity-скрипты, триггеры, AI, кастомную интерактивность и всю игровую логику в v1.
- Не пытаться сразу поддержать все сцены SenseTower массово.
- Не делать универсальный scene format под все возможные движки и рантаймы.
- Не добиваться визуального 1:1 паритета с Unity по свету, шейдерам и пост-эффектам в первом этапе.

## Предпосылки и ограничения

- Текущий web runtime строит сцену напрямую в `apps/runtime-web/src/main.ts`; отдельного scene-loader слоя пока нет.
- Точка входа room boot уже есть: manifest приходит из `apps/api/src/index.ts`.
- В проекте уже есть зачатки manifest/template подхода: `packages/templates/src/index.ts`, `packages/templates/src/registry.ts`.
- Текущий asset pipeline понимает web-friendly форматы (`.glb`, `.gltf`, `.ktx2`), но пока не умеет сценовый импорт из Unity: `packages/asset-pipeline`, `docs/asset-pipeline.md`.
- Источники сцен находятся вне репозитория `noah` и выглядят как Unity-проекты с `.unity` сценами и `.fbx` ассетами: `/mnt/d/sensetowervr`, `/mnt/d/repository/_SenseCapital/SenseTowerVR`.
- Для spike v1 выбирается один конкретный и повторяемый способ экспорта из Unity в `.glb`; смена экспортера в рамках v1 не планируется.
- В v1 целевая совместимость только для web runtime `noah`; Unity остается source of truth для исходных сцен.
- Первая поставка - короткий spike на 1-2 недели, поэтому основной deliverable - работающий pipeline на 1-2 сценах, а не массовая миграция.
- В первую версию входят статика, материалы/текстуры по возможности, точки входа и room metadata; перенос интерактивов и сложной навигации откладывается.
- Scene bundle в v1 публикуется как static asset, доступный runtime по URL; room manifest хранит ссылку на него как опциональное поле.
- `scene.json` использует `schemaVersion: 1`; если runtime встречает неизвестную версию схемы, он обязан уйти в fallback-сцену и показать диагностику.
- Целевые пилотные кандидаты для spike: один hall из `/mnt/d/sensetowervr/Assets/Scenes/TheHallScene.unity` или `/mnt/d/sensetowervr/Assets/Scenes/TheHallScene2.unity` и одна сцена попроще вроде `/mnt/d/sensetowervr/Assets/Scenes/TheOfficeScene.unity`.
- Для v1 закладываются грубые performance-рамки: bundle одной сцены должен иметь зафиксированный размерный бюджет, cold load должен быть измерен на desktop, а mobile проверяется как smoke/acceptability, не как strict parity.

## Подход

Взять самый простой жизнеспособный путь:

1. Зафиксировать минимальный Scene Bundle format для `noah`: один основной `scene.glb` плюс JSON-манифест с метаданными комнаты.
2. Сначала подтвердить экспорт одной конкретной сцены из Unity в этот bundle, не строя сразу сложный универсальный экспортер.
3. Только после успешного экспорта первой сцены добавить в `noah` scene-loader и поддержку room manifest, который выбирает scene bundle для конкретной комнаты.
4. Затем распространить тот же путь на вторую пилотную сцену и проверить, что разные комнаты действительно открывают разные bundle.
5. После подтверждения pipeline уже решать, стоит ли автоматизировать bulk-экспорт и расширять формат.

## Задачи

### Iteration 1 - зафиксировать формат и выбрать пилотные сцены

- [ ] Утвердить 2 пилотные сцены для v1: один hall и одну комнату попроще.
- [ ] Описать минимальный Scene Bundle format в `docs/`: `scene.glb` + `scene.json`.
- [ ] Зафиксировать обязательные поля `scene.json`: `schemaVersion`, `sceneId`, `label`, `source`, `glbPath`, `spawnPoints`, `bounds`, `preview`, `notes`.
- [ ] Разделить поля `scene.json` на обязательные и опциональные и описать поведение runtime при отсутствии каждого опционального поля.
- [ ] Явно зафиксировать, что в v1 формат не содержит Unity behavior/scripts и не обещает перенос baked lighting один-в-один.
- [ ] Утвердить место публикации bundle для v1 и правило именования каталогов/файлов.
- [ ] Зафиксировать naming convention для spawn points и способ их задания: вручную в `scene.json` или через выделенные объекты/точки в Unity.
- [ ] Зафиксировать размерный бюджет bundle и минимальные метрики приемки загрузки для desktop smoke.

### Iteration 2 - подтвердить экспорт из Unity

- [ ] Найти и задокументировать фактический source path для выбранных сцен в `/mnt/d/sensetowervr` или `/mnt/d/repository/_SenseCapital/SenseTowerVR`.
- [ ] Выбрать один конкретный exporter/toolchain для Unity -> `.glb` и зафиксировать его как единственный путь для spike v1.
- [ ] Для каждой пилотной сцены определить состав экспорта: основной mesh, зависимые текстуры, отдельные проблемные материалы, точки входа.
- [ ] Сделать рабочий экспорт первой пилотной сцены в `.glb` выбранным toolchain.
- [ ] Проверить, что первая экспортированная сцена открывается как отдельный bundle и содержит ожидаемые базовые данные.
- [ ] Сделать рабочий экспорт второй пилотной сцены тем же toolchain без изменения формата.
- [ ] Сохранить рядом с экспортом человекочитаемую инструкцию, как повторить экспорт той же сцены.
- [ ] Проверить размер итоговых `.glb` и зафиксировать бюджет/ограничения для web runtime.
- [ ] Если часть материалов или объектов не экспортируется корректно, задокументировать допустимые упрощения v1 и список известных потерь.

### Iteration 3 - подключить scene bundle в noah

- [ ] Добавить типы scene bundle и новое опциональное поле manifest в `apps/runtime-web/src/index.ts` и связанных типах API.
- [ ] Добавить loader для `scene.json` с валидацией обязательных полей и `schemaVersion`.
- [ ] Добавить loader для `scene.glb` в runtime и определить его интеграцию с текущим boot flow.
- [ ] Применить из bundle базовые метаданные сцены: spawn point, bounds, display label.
- [ ] Сохранить текущую fallback-сцену, если bundle не задан, невалиден или не загрузился.
- [ ] Расширить room manifest/API так, чтобы комната могла ссылаться на scene bundle без breaking changes для существующих комнат.
- [ ] Обновить template/registry слой так, чтобы шаблон мог либо использовать дефолтную сцену, либо переопределяться scene bundle'ом комнаты.
- [ ] Создать 2 тестовые комнаты, где каждая открывает свою экспортированную сцену из SenseTower.
- [ ] Добавить feature flag или env toggle для быстрого отключения нового scene bundle path.

### Iteration 4 - минимальная операционализация и подготовка к масштабированию

- [ ] Добавить валидацию scene bundle: наличие `scene.json`, `scene.glb`, корректность обязательных полей, валидность ссылок.
- [ ] Зафиксировать чек-лист готовности новой сцены к подключению в room.
- [ ] Описать процесс добавления третьей сцены без изменений в коде runtime, только через экспорт + регистрацию manifest/room config.
- [ ] Зафиксировать решение, какие части pipeline пока остаются ручными, а какие стоит автоматизировать на следующем этапе.
- [ ] Сформировать backlog для phase 2: bulk export, nav/colliders, light probes, интерактивы, оптимизация материалов, CDN/storage.

## Тест-план

- **Unit**
- [ ] Проверка парсинга `scene.json` и обязательных полей bundle.
- [ ] Проверка room manifest mapping: комната без scene bundle продолжает работать по старому пути.
- [ ] Проверка выбора spawn point и fallback-значений при неполных метаданных.
- [ ] Проверка, что неизвестная `schemaVersion` переводит runtime в fallback.
- [ ] Проверка feature flag/env toggle для отключения нового scene bundle path.

- **Integration**
- [ ] Runtime успешно загружает валидный `scene.glb` + `scene.json` и инициализирует сцену без падения boot flow.
- [ ] API отдает расширенный manifest со scene bundle обратно-совместимо для существующих клиентов.
- [ ] Комната A и комната B открывают разные сцены без ручной смены кода в runtime.
- [ ] Bundle с отсутствующим `scene.glb` или битым `scene.json` переводит runtime в fallback-сцену с понятной диагностикой.
- [ ] Комната без нового поля scene bundle работает без изменений по старому template path.

- **E2E**
- [ ] Smoke: открыть room на первой экспортированной сцене и убедиться, что runtime загружается и пользователь может войти в комнату.
- [ ] Smoke: открыть вторую room и убедиться, что загружается другая сцена.
- [ ] Negative: отсутствующий scene bundle не ломает room flow, а использует fallback.
- [ ] Negative: слишком большой или медленно грузящийся bundle показывает понятный статус загрузки/ошибки и не ломает страницу.

- **Manual QA**
- [ ] Сверить пилотные сцены визуально с Unity-источником по базовой геометрии, ориентации, масштабу и точке входа.
- [ ] Проверить desktop/mobile браузеры на приемлемое время загрузки и FPS на пилотных сценах.
- [ ] Проверить, что разные комнаты действительно маппятся на разные сцены SenseTower.

- **CI / automation boundary**
- [ ] Все unit/integration проверки scene bundle и manifest compatibility выполняются в CI без доступа к Unity-проекту.
- [ ] Проверка самого Unity export pipeline в spike остается manual и подтверждается сохраненными артефактами/инструкцией.

## Риски и откаты (roll-back)

- Риск: попытка тащить `.unity` напрямую или поддерживать слишком общий формат раздует spike.
  - Откат: ограничиться Scene Bundle format поверх `.glb` + JSON и 1-2 сценами.
- Риск: экспорт из Unity окажется неполным по материалам/свету.
  - Откат: принять упрощенный визуальный профиль v1 и документировать потери явно.
- Риск: большие сцены окажутся слишком тяжелыми для web runtime.
  - Откат: брать пилотные сцены умеренного размера, резать качество, дробить экспорт или временно исключать тяжелые объекты.
- Риск: изменения в `main.ts` сделают runtime еще более монолитным.
  - Откат: выносить scene-loader и bundle parsing в отдельные модули, сохранить fallback-сцену.
- Риск: room manifest changes сломают текущие комнаты.
  - Откат: добавлять scene bundle как опциональное поле, при отсутствии использовать текущий template-driven path.
- Риск: выбранный Unity exporter не справится с одной из нужных сцен.
  - Откат: подтвердить pipeline на более простой сцене, оставить сложную сцену в backlog и не расширять scope v1.
