# ADR — интерактивные медийные поверхности и медийные объекты

## Статус

Предложено для фазы M1-MEDIA.

## Контекст

VRATA переходит от базовой многопользовательской комнаты к деловому сценарию: встречи, презентации, демонстрации, доска, несколько поверхностей и управление из виртуальной реальности.

Нужно добавить возможности:

- трансляция экрана ведущего;
- трансляция веб-содержимого ведущего как видеопотока;
- интерактивная доска;
- несколько поверхностей в одной комнате;
- ввод лучом из виртуальной реальности;
- будущие медийные объекты от независимых разработчиков.

Главная опасность: встроить каждую новую возможность напрямую в ядро среды исполнения.

## Решение

Ввести архитектуру медийных объектов:

1. Медийная поверхность — объект сцены, который отображает содержимое и принимает ввод.
2. Медийный объект — независимая функциональная сущность, прикрепляемая к поверхности.
3. Протокол ввода поверхности — единый формат событий мыши, касания, клавиатуры и VR-луча.
4. Реестр медийных объектов — механизм регистрации типов объектов.
5. Протокол расширений — формальный способ добавления новых объектов без изменения ядра.

## Принятые различия

### Трансляция экрана

Трансляция экрана является видеопотоком.

Она не дает прямого управления страницей, находящейся внутри потока.

### Управляемая веб-поверхность

Управляемая веб-поверхность требует исполнителя команд: удаленного браузера или специальной интеграции.

Она является отдельным объектом.

### Доска

Доска является первым собственным интерактивным объектом VRATA и используется для проверки протокола ввода.

## Последствия

Положительные:

- новые объекты можно добавлять через общий протокол;
- ввод из VR, мыши и касания становится единым;
- доска, трансляция и удаленный браузер не смешиваются;
- несколько поверхностей работают независимо;
- будущие независимые разработчики получают понятную модель расширений.

Отрицательные:

- архитектура сложнее, чем прямое добавление одной кнопки трансляции;
- нужно больше типов и автоматических проверок;
- удаленный браузерный объект требует отдельного исполнителя и политики безопасности.

## Альтернативы

### Альтернатива 1. Встроить трансляцию и доску напрямую в ядро

Отклонено.

Причина: быстрый старт, но плохая расширяемость. Каждый новый объект будет требовать изменения ядра.

### Альтернатива 2. Сразу открыть произвольные внешние расширения

Отклонено для M1-MEDIA.

Причина: слишком много вопросов безопасности, изоляции и версионирования. В M1-MEDIA расширения должны быть внутренними, но использовать будущий протокол.

### Альтернатива 3. Считать трансляцию экрана управляемым веб-объектом

Отклонено.

Причина: видеопоток и управляемая страница — разные сущности. Для настоящего управления страницей нужен удаленный браузерный объект или специальная интеграция.

## Критерии правильной реализации

- Объект можно создать на поверхности через состояние комнаты.
- Объект получает ввод через `SurfaceInputEvent`.
- Объект объявляет возможности и разрешения.
- Несколько объектов на разных поверхностях не смешивают состояние.
- Новое внутреннее расширение можно добавить без изменения ядра.
- Автоматические проверки подтверждают роли, ввод, поверхности, объекты и приемочный сценарий.
