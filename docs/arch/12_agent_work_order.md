# Рабочее задание агенту: M1-MEDIA

## Роль агента

Ты работаешь как инженер-архитектор и ведущий разработчик фазы M1-MEDIA.

Твоя задача — добавить интерактивные медийные объекты в VRATA так, чтобы они были расширяемыми и не ломали уже закрытую основу M0.5.

## Стартовый контекст

M0.5 считается закрытой. Не возвращайся к базовой приемке присутствия, если новая работа ее не ломает.

M1.1, M1.2, M1.3 и M1.4 считаются закрытыми 2026-05-12. Post-M1.4 per-surface media audio policy считается закрытой 2026-05-13. M1.5 считается закрытой 2026-05-14. M1.6 пропущена 2026-05-14, потому что сценарий уже закрыт M1.4/post-M1.4 `screen-share`. M1.7 принят 2026-05-21 как полноценный `remote-browser` executor with LiveKit viewport stream. M1.8 принят 2026-05-21 как несколько независимых медийных поверхностей в одной комнате. M1.9 принят 2026-06-10 как internal extension protocol with registry, capability/permission validation, debug snapshot, and focused staging verification. Следующая подфаза: `M1.10 — приемка и выпуск M1-MEDIA`.

Уже есть:

- вход в комнату;
- базовое присутствие;
- режим обычного браузера, телефона и виртуальной реальности;
- голос и медиа-диагностика;
- отладочные данные;
- автоматические проверки;
- проверочная среда.

## Главная цель

Создать систему, где ведущий может:

1. авторизоваться как ведущий;
2. выбрать медийную поверхность;
3. запустить трансляцию экрана на одной поверхности;
4. запустить доску на другой поверхности;
5. управлять поверхностями мышью, касанием и лучом из виртуальной реальности;
6. использовать несколько поверхностей одновременно;
7. подключать будущие медийные объекты через протокол расширений.

## Ключевой архитектурный запрет

Не вшивай каждый объект в ядро среды исполнения.

Правильный путь:

```text
ядро среды исполнения
        ↓
протокол поверхности
        ↓
реестр медийных объектов
        ↓
медийный объект
```

Неправильный путь:

```text
ядро среды исполнения напрямую знает доску, трансляцию, браузер и каждую будущую функцию
```

## Термины, которые использовать

Используй слово **протокол**.

Для технических интерфейсов всегда используй термин «протокол».

Предпочтительные термины:

- протокол ввода поверхности;
- протокол медийных объектов;
- протокол расширений;
- медийная поверхность;
- медийный объект;
- среда исполнения;
- служба состояния комнаты;
- медиа-транспорт;
- ведущий;
- участник;
- гость.

### Шаг 7. Трансляция веб-содержимого ведущего

Статус: `skipped`.

Не выполняй подфазу `M1.6` как отдельную реализацию: она дублирует уже закрытый `screen-share` из M1.4/post-M1.4.

Критерий готовности:

- пользовательский сценарий показа вкладки, окна или экрана закрыт через `screen-share`;
- система не обещает прямое управление страницей там, где идет только видеопоток;
- отдельные тесты `host-web-broadcast` не требуются, чтобы не закреплять дублирующий объект.

### Шаг 8. Удаленный браузерный объект

Статус: `done`.

Подфаза `M1.7` закрыта как полноценный remote-browser executor service, а не mock/prototype. Объем остается безопасно ограниченным allowlisted domains/test pages; backend executor, LiveKit viewport stream, command flow, lifecycle cleanup and staging deploy выполнены.

Критерий готовности:

- есть объект `remote-browser` и отдельный backend executor service;
- ввод проходит через протокол поверхности;
- запрещенные адреса отклоняются;
- реальный Chromium-rendered result stream виден на поверхности;
- tests `remote-browser-object` and staging gate проходят локально и на staging.

Принятое ограничение: аудио на VR и мобильных устройствах может подтормаживать; это QoS/performance follow-up и не блокирует M1.8, пока не ломает базовый сценарий медийных поверхностей.

### Шаг 11. Единая приемка

Добавь команду:

```bash
pnpm run test:m1-media
```

Критерий готовности:

- все стандартные проверки проекта проходят;
- приемочный тест `m1-media-acceptance` проходит.

### Шаг 12. Финальная ручная проверка

Провести только после зеленых автоматических проверок.

Проверить:

- реальный выбор вкладки, окна или экрана;
- реальное устройство виртуальной реальности;
- выбор поверхности лучом;
- одновременную работу доски и трансляции;
- выход и повторный вход участника.

## Общие критерии качества

1. Не ломать M0.5.
2. Не переписывать медиа-транспорт.
3. Не смешивать состояние комнаты и медиа-транспорт.
4. Не делать отдельный путь только для VR.
5. Не называть видеотрансляцию управляемым браузером.
6. Не использовать ручную проверку там, где можно написать автоматическую.
7. Документировать ограничения сразу.
