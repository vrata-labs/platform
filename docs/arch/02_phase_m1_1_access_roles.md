## Цель

Ввести минимальную авторизацию и роль ведущего, чтобы управление медийными объектами было предсказуемым.

Без этой подфазы нельзя корректно реализовать показ экрана, доску и удаленный браузерный объект: система должна знать, кто имеет право создавать объект, кто может рисовать, кто может останавливать трансляцию и кто управляет поверхностью.

## Область подфазы

Входит:

- выдача участнику роли при входе;
- проверка прав на действия с поверхностями;
- отображение роли в отладочных данных;
- минимальная защита действий ведущего;
- автоматические проверки прав.

Не входит:

- полноценная учетная запись организации;
- сложная панель управления пользователями;
- интеграция с внешними поставщиками единого входа;
- платные тарифы;
- сложные политики групп.

## Термины

### Роль

Роль — это назначенный участнику уровень полномочий в комнате.

Минимальные роли:

- `guest` — гость, может войти, смотреть поверхности и подключать голос;
- `member` — участник, может взаимодействовать с разрешенными объектами, например рисовать на доске;
- `host` — ведущий, может запускать и останавливать медийные объекты;
- `admin` — администратор, может управлять комнатой и политиками.

### Разрешение

Разрешение — конкретное действие, которое роль может выполнять. Например: `screen-share.start`, `whiteboard.clear`, `surface.lock`.

## Минимальная схема токена

```ts
type RoomAccessTokenPayload = {
  tenantId?: string;
  roomId: string;
  participantId: string;
  displayName: string;
  role: 'guest' | 'member' | 'host' | 'admin';
  permissions: string[];
  exp: number;
};
```

Если полноценная подпись токена уже реализована, расширить существующий токен. Если нет — сделать минимальную схему, совместимую с будущей полноценной авторизацией.

## Решение M1.1

M1.1 вводит фундамент доступа: роль участника, вычисляемые разрешения, отладочный контракт и host-only gating для уже существующих ведущих действий.

M1.1 не создает полноценные медийные поверхности и медийные объекты. Разрешения `surface.create-object`, `surface.stop-object` и `surface.lock` добавляются в политику как будущие значения, но серверные проверки конкретных surface-команд реализуются в M1.3, когда появится протокол поверхностей.

## Источник истины роли

- по умолчанию участник входит как `guest`;
- в dev/staging допускается `?role=host`, `?role=member` или `?role=admin`, но только если включен dev/staging gate;
- runtime не может сам повысить свою роль через `participant_update` или другое клиентское состояние;
- служба состояния комнаты должна получать роль из серверно принятого access context/token или из явно разрешенного dev/staging query path;
- при ошибке разбора роли используется безопасный fallback `guest`.

## Минимальная матрица разрешений

- `guest`: `room.join`, `audio.join`, `surface.view`;
- `member`: `room.join`, `audio.join`, `surface.view`, `surface.input`, `whiteboard.draw`;
- `host`: `room.join`, `audio.join`, `surface.view`, `surface.select`, `surface.create-object`, `surface.stop-object`, `surface.lock`, `surface.input`, `screen-share.start`, `screen-share.stop`, `whiteboard.draw`, `whiteboard.clear`;
- `admin`: все разрешения `host` плюс `room.admin`.

```ts

  role: 'guest' | 'member' | 'host' | 'admin';

  canControlSurface: boolean;
```


Этот способ допустим только в режиме разработки или проверочной среды. Для боевой среды он должен быть заменен серверной выдачей роли.

## Автоматические проверки

Создать файл:

```text
tests/e2e/m1-media/access-roles.spec.ts
```

Проверки:

1. Гость входит в комнату и имеет роль `guest`.
2. Ведущий входит через разрешенный dev/staging путь и имеет роль `host`.
3. Гость не видит или не может нажать кнопку запуска трансляции экрана.
4. Ведущий видит и может нажать кнопку запуска трансляции экрана.
5. Клиент не может повысить роль через `participant_update`.
6. Служба состояния комнаты отклоняет privileged command от `guest` в тестовом auth-gated message handler.
7. `window.__VRATA_DEBUG__.access` содержит роль, permissions и derived booleans.

## Критерии готовности

Подфаза закрыта, если:

- у каждого участника есть роль;
- права действий вычисляются явно;
- ключевые действия с поверхностями проверяются на серверной стороне;
- интерфейс не показывает ведущие действия гостю;
- автоматические проверки прав проходят;
- обычный вход в комнату не сломан.

Фактический результат M1.1: критерии закрыты. Реальные медийные поверхности и surface-команды остаются за M1.2/M1.3, как было задано границами подфазы.

## Что не считать готовностью

- Роль есть только в клиентском интерфейсе, но не проверяется службой состояния комнаты.
- Гость может отправить команду напрямую и создать объект.
- Ведущий работает только через жестко зашитый идентификатор участника.
- Проверка есть только ручная.
