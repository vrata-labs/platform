# M1.8 — несколько медийных поверхностей и схемы размещения

## Статус

Статус: `done` с 2026-05-21.

M1.8 начинается после принятого M1.7 `remote-browser`. Известные QoS-проблемы аудио на VR/мобиле остаются отдельным follow-up и не должны блокировать работу над несколькими поверхностями, если не ломают базовый сценарий surface rendering/input.

## Результат реализации

Первый slice M1.8 реализован и проверен на staging:

- состояние комнаты теперь содержит три независимые поверхности: `debug-main`, `whiteboard-wall`, `laptop-screen`;
- runtime показывает выбор поверхности в HUD;
- создание, остановка, ввод и текстуры объектов маршрутизируются по выбранной или фактически задетой поверхности;
- `screen-share`, `whiteboard` и `remote-browser` больше не делят одну глобальную поверхность;
- остановка объекта на одной поверхности не очищает объект на другой поверхности.

## Цель

Поддержать несколько независимых медийных поверхностей в одной комнате.

Нужно, чтобы ведущий мог, например:

- включить доску на одной поверхности;
- включить презентацию или трансляцию экрана на другой;
- оставить третью поверхность для будущего объекта;
- выбрать поверхность в VR лучом;
- управлять объектами независимо.

## Схема комнаты

В описании комнаты добавить список поверхностей:

```ts
type RoomManifestMedia = {
  surfaces: Array<{
    surfaceId: string;
    label: string;
    kind: 'wall' | 'table' | 'laptop' | 'floating' | 'custom';
    transform: {
      x: number;
      y: number;
      z: number;
      yaw: number;
      pitch: number;
      roll: number;
    };
    size: {
      widthM: number;
      heightM: number;
    };
    defaultObjectType?: string;
    allowedObjectTypes: string[];
  }>;
};
```

## Примеры поверхностей

### Главная стена

Большая поверхность для презентации или трансляции экрана.

### Доска

Поверхность сбоку или на отдельной стене для рисования.

### Виртуальный ноут

Малая поверхность на 3D-модели ноутбука. Может показывать поток ведущего или удаленный браузер.

### Плавающая поверхность

Поверхность, которую ведущий может расположить ближе к участникам. В первой версии можно сделать статичной.

## Правила независимости

1. Каждая поверхность имеет собственный `surfaceId`.
2. Каждая поверхность имеет не более одного активного основного объекта.
3. Один объект может зеркально отображаться на нескольких поверхностях только через явное поле `mirrorSurfaceIds`.
4. Ввод направляется только в активный объект выбранной поверхности.
5. Остановка объекта на одной поверхности не должна очищать другую поверхность.
6. Роль ведущего действует на все поверхности, но блокировка управления может быть отдельной для каждой.

## Пользовательский сценарий

1. Ведущий выбирает главную стену.
2. Запускает трансляцию экрана.
3. Выбирает боковую поверхность.
4. Запускает доску.
5. Рисует на доске лучом из VR.
6. Другие участники видят и трансляцию, и доску.
7. Ведущий останавливает только трансляцию; доска остается.
8. Ведущий очищает доску; трансляция не затрагивается.

## Задачи агента

1. Добавить список поверхностей в описание комнаты.
2. Перенести существующую поверхность в эту схему.
3. Добавить минимум две поверхности в базовую комнату:
   - `main-screen`;
   - `whiteboard-wall`.
4. При возможности добавить третью поверхность:
   - `laptop-screen`.
5. Добавить выбор поверхности мышью и лучом.
6. Добавить меню выбранной поверхности.
7. Убедиться, что активные объекты не смешиваются.
8. Добавить отладочные данные по всем поверхностям.

## Автоматические проверки

Создать:

```text
tests/e2e/m1-media/multi-surface-layouts.spec.ts
```

Проверки:

1. В комнате есть минимум две поверхности.
2. Ведущий создает `screen-share` на `main-screen`.
3. Ведущий создает `whiteboard` на `whiteboard-wall`.
4. Участник видит два разных активных объекта.
5. Клик по доске не меняет состояние трансляции.
6. Остановка трансляции не очищает доску.
7. Очистка доски не останавливает трансляцию.
8. Имитация луча виртуальной реальности выбирает нужную поверхность.
9. Отладочные данные показывают правильные `surfaceId` и `objectId`.

## Критерии готовности

Подфаза закрыта, если:

- в комнате есть несколько независимых поверхностей;
- на разных поверхностях одновременно работают разные объекты;
- выбор поверхности работает в обычном браузере и через имитацию VR-ввода;
- состояние объектов не смешивается;
- автоматические проверки проходят.

## Что не считать готовностью

- Есть две визуальные плоскости, но они используют один общий объект состояния.
- Доска и трансляция не могут работать одновременно.
- Остановка одного объекта очищает все поверхности.
- Выбор поверхности доступен только мышью, но не подготовлен к VR-лучу.
