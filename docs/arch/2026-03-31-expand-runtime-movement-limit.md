# План: расширить доступную для перемещения сцену во все стороны

## Цель

Увеличить доступную область перемещения пользователя в runtime web/VR/mobile с текущего глобального лимита `6` до `24` по `x/z`, не меняя скорость, spawn-логику, scene bundle contract и остальное поведение runtime.

## Не-цель

- Не вводить scene-specific bounds или настройку лимита через `scene.json`.
- Не менять контракты `apps/runtime-web/src/index.ts` и `apps/runtime-web/src/scene-bundle.ts`.
- Не менять скорость движения, snap turn, touch/XR input и spawn point поведение.
- Не добавлять новую конфигурацию, feature flag или admin/runtime API, если достаточно правки текущего глобального clamp.
- Не решать navmesh/collision/teleport; задача ограничена только расширением доступной области свободного перемещения.

## Предпосылки и ограничения

- Текущее ограничение перемещения находится в `apps/runtime-web/src/movement.ts`: `clampRoomPosition(position, limit = 6)`; именно оно зажимает движение по `x/z` в квадрат `[-6, 6]`.
- `updateMovement()` в `apps/runtime-web/src/main.ts` использует `stepFlatMovement()` и не задает свой limit, значит глобальная смена default limit затронет desktop, mobile touch, bot mode и VR path одинаково.
- В репозитории уже есть unit-тесты для movement-утилит в `apps/runtime-web/src/movement.test.ts`.
- В репозитории уже есть Playwright runtime e2e в `tests/e2e/runtime.spec.ts`; для этой задачи достаточно расширить существующий набор, а не заводить новый test stack.
- Пользователь явно выбрал простой путь: глобальный лимит для всех комнат, с сохранением обратной совместимости по остальному поведению.
- Отдельная защита от слишком большого лимита, новая конфигурация и staging rollout в рамках этой задачи не требуются, если локальные проверки достаточны.

## Подход

Сделать минимальное изменение в одном месте: увеличить текущий default limit в `clampRoomPosition()` с `6` до `24`, затем закрепить это unit-тестом и e2e-сценарием. Для e2e использовать уже существующий bot movement path, чтобы детерминированно подтвердить, что локальная позиция уходит дальше прежней границы `6` по одной из осей, а runtime при этом остается работоспособным.

## Задачи

- [ ] Обновить default limit в `clampRoomPosition()` с `6` до `24` без изменения сигнатур публичных helper'ов и без изменения логики normalizing/rotation/movement speed.
- [ ] Проверить, что `stepFlatMovement()` продолжает использовать тот же clamp path и что desktop/mobile/VR ветки `updateMovement()` не требуют дополнительных изменений в runtime-коде.
- [ ] Обновить unit-тесты в `apps/runtime-web/src/movement.test.ts`, чтобы они проверяли прохождение координат внутри нового лимита `24` и корректный clamp на границе `-24..24`.
- [ ] Расширить e2e-сценарий в `tests/e2e/runtime.spec.ts`, используя bot movement path, чтобы детерминированно подтвердить выход локальной позиции дальше старого предела `6` по `x` или `z`.
- [ ] Прогнать локальные проверки `pnpm test` и `pnpm test:e2e`, убедиться, что существующие runtime/presence сценарии не деградировали.
- [ ] Зафиксировать только при необходимости найденные зависимости от старой границы как отдельный follow-up, не раздувая scope текущего изменения.

## Тест-план

- **Unit**
- [ ] `clampRoomPosition()` допускает координаты внутри нового глобального предела `24`, даже если они больше старого лимита `6`.
- [ ] `clampRoomPosition()` по-прежнему ограничивает `x` и `z` на симметричной границе `-24..24`.
- [ ] `stepFlatMovement()` сохраняет прежнюю нормализацию диагонального движения и не меняет поведение скорости.

- **E2E**
- [ ] Runtime room успешно загружается как и раньше: HUD виден, статус joined/presence остается рабочим.
- [ ] Сценарий с bot movement path уводит локального игрока дальше прежнего предела `6` по одной из осей.
- [ ] После перемещения за старую границу runtime не падает, а debug state/presence продолжают обновляться.

- **Негативные кейсы**
- [ ] Попытка уйти дальше нового лимита все еще приводит к clamp, а не к бесконечному выходу координат.
- [ ] Диагональное движение возле новой границы не ломает clamp по одной из осей.
- [ ] Bot mode и обычный ручной movement path продолжают работать через тот же helper без расхождения по логике.

## Риски и откаты (roll-back)

- Риск: слишком большой глобальный лимит позволит уходить в визуально пустые зоны fallback-комнаты или за пределы ожидаемого scene framing.
  - Откат: вернуть прежнее значение лимита одним изменением в `apps/runtime-web/src/movement.ts`.
- Риск: существующие e2e или runtime assumptions неявно завязаны на старый предел и начнут падать.
  - Откат: скорректировать лимит до более умеренного значения или временно вернуть старое до выяснения зависимости.
- Риск: в отдельных сценах большой радиус движения ухудшит UX из-за отсутствия коллизий и навигационных ограничений.
  - Откат: оставить глобальное изменение только как временное и в следующей итерации перейти на scene-specific bounds, если это действительно потребуется.
