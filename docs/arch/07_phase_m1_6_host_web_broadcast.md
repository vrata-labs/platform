# M1.6 — трансляция веб-содержимого ведущего

## Итог решения

Статус: `skipped` по состоянию на 2026-05-14.

Причина: эта подфаза дублирует уже закрытый M1.4/post-M1.4 `screen-share`. Пользовательский сценарий «ведущий показывает вкладку, окно, экран или сайт, открытый у себя» уже реализован как видеопоток ведущего на медийной поверхности.

Что считается закрытым M1.4/post-M1.4:

- `screen-share` работает как медийный объект;
- поток привязан к выбранной медийной поверхности;
- host/admin права проверяются;
- зрители видят видеопоток на поверхности;
- аудио-политика поверхности вынесена в post-M1.4;
- автоматические и staging-проверки `screen-share-object` пройдены.

Что остается не покрытым и не должно называться M1.6:

- клик по произвольной веб-странице из VR;
- ввод текста, скролл или навигация внутри удаленной страницы;
- открытие URL внутри VRATA отдельным исполнителем.

Эти сценарии относятся к M1.7 `remote-browser`.

## Цель

Описать и реализовать сценарий, в котором ведущий показывает на медийной поверхности любой веб-контент, который он сам открыл: Google-презентацию, YouTube, сайт, документ или внутреннюю страницу.

Это не отдельный способ читать веб-страницы. Это пользовательский сценарий поверх объекта трансляции экрана.

## Главная формулировка

Ведущий показывает не «сайт как встроенный документ», а **свой видеопоток экрана, окна или вкладки**.

Поэтому для VRATA содержимое является видеопотоком.

```text
ведущий открывает адрес у себя
        ↓
ведущий выбирает окно или вкладку для захвата
        ↓
VRATA получает видеопоток
        ↓
поток выводится на медийную поверхность
        ↓
участники смотрят
```

## Что это решает

Этот сценарий позволяет быстро показывать:

- презентации;
- видео;
- сайты;
- документы;
- демонстрацию интерфейса продукта;
- внутренние страницы, доступные только ведущему.

## Что это не решает

Этот сценарий сам по себе не дает участникам возможности кликать кнопки внутри транслируемой страницы.

Причина простая: в комнате находится видеокопия. Настоящая страница открыта у ведущего. Клик по видеокопии не равен клику в отдельной вкладке ведущего.

Если нужно именно управление страницей с поверхности, использовать подфазу `M1.7 — удаленный браузерный объект`.

## Сценарий с виртуальным ноутом

Виртуальный ноут в комнате может быть реализован как:

```text
3D-модель ноутбука
        +
медийная поверхность вместо экрана
        +
объект трансляции экрана ведущего
```

Тогда ведущий и остальные участники видят поток на экране виртуального ноутбука.

Если нужно, чтобы лучом в VR можно было нажимать кнопки внутри страницы на ноутбуке, ноут должен использовать удаленный браузерный объект или специальную интеграцию.

## Потоки управления

### Вариант А. Ведущий управляет контентом вне VRATA

1. Ведущий открывает вкладку или окно.
2. Ведущий запускает трансляцию.
3. Ведущий управляет страницей обычной мышью, клавиатурой или средствами своей системы.
4. Остальные видят результат.

Это первый и самый простой вариант.

### Вариант Б. Ведущий в VR запускает показ заранее выбранного окна или вкладки

1. Ведущий выбирает поверхность в VR.
2. Нажимает «показать вкладку».
3. Браузер может вывести системный выбор источника захвата.
4. После выбора поток отображается на поверхности.

Реальное поведение зависит от браузера и устройства. Поэтому этот сценарий требует ручной проверки.

### Вариант В. Ведущий в VR управляет настоящей страницей лучом

Это не трансляция экрана. Это удаленный браузерный объект.

## Задачи агента

1. Добавить в интерфейс ведущего отдельную формулировку: «Показать вкладку, окно или экран».
2. Для поверхности добавить состояние «ожидает выбора источника».
3. Добавить подсказку зрителям: «Ведущий готовит показ».
4. Добавить режим виртуального ноутбука как вариант размещения поверхности, если модель или сцена это поддерживает.
5. Добавить документацию, что это видеопоток, а не управляемый веб-документ.
6. Добавить проверку, что неизвестный веб-адрес не передается в VRATA как обязательный параметр для чтения.

## Автоматические проверки

Создать:

```text
tests/e2e/m1-media/host-web-broadcast.spec.ts
```

Проверки:

1. Ведущий может выбрать поверхность и перейти в состояние подготовки трансляции.
2. Состояние подготовки видно другим участникам.
3. После подмененной успешной публикации объект становится активным.
4. Состояние явно указывает тип `host-web-broadcast` или `screen-share` в зависимости от выбранной реализации.
5. Система не пытается загрузить внешний URL как содержимое ядра среды исполнения.
6. Гость не может начать такой показ.

## Критерии готовности

Подфаза закрыта, если:

- сценарий показа любого веб-содержимого ведущего описан и реализован как поток;
- ведущий может показать вкладку, окно или экран на выбранной поверхности;
- зрители видят поток;
- интерфейс не обещает клик по кнопкам страницы там, где это только видеопоток;
- автоматические проверки подтверждают состояние подготовки, активности и остановки.

## Что не считать готовностью

- Пользовательский интерфейс говорит «открой любой сайт на поверхности», хотя на самом деле это только видеопоток.
- Система пытается загрузить произвольный URL внутрь сцены без отдельного исполнителя.
- Ведущий в VR может выбрать поверхность, но не видит состояния подготовки.
- Поток не связан с конкретной поверхностью.
