# M0.5 Presence Acceptance

## Goal

Close the technical baseline for multi-user presence in a spatial web room.

This phase validates reliability, not avatar richness. A participant must be able to enter a room, be visible to others, move, expose approximate facing direction, control microphone state explicitly, and leave/rejoin without stale visual or audio remnants.

## Basic Presence

Basic presence means:

- participants join the same room by URL;
- every participant sees the other active participants;
- remote participant movement is visible and smoothed;
- remote participant root and head orientation are propagated;
- voice starts only after explicit user action;
- microphone state is visible in room diagnostics and remote participant state;
- spatial audio source diagnostics follow participant pose;
- leaving and rejoining does not leave stale participants;
- browser, mobile, and VR-mode presence use the same room link and state layer.

## Out Of Scope

M0.5 explicitly does not include:

- humanoid avatars, skeletons, fingers, facial animation, clothing, or avatar customization;
- new room templates or scene migration work;
- a full control-plane expansion;
- rewriting the media layer or replacing the current 3D engine;
- a separate runtime path that works only in VR.

## Automatic Acceptance

The phase is accepted automatically when tests cover:

- stable `window.__NOAH_DEBUG__` presence, media, and spatial-audio contract;
- pose propagation with root/head position and orientation;
- deterministic bot participants for idle, line, turn, and square motion;
- four browser participants in one room, each seeing three remotes;
- join, leave, and rejoin cleanup without stale remotes;
- movement smoothing diagnostics with update frequency, staleness, and jump limits;
- voice state transitions, microphone mute propagation, and microphone denial degraded mode;
- spatial-audio listener/source diagnostics and `?spatial=0` fallback;
- `?xrmock=1` VR presence simulation without a physical device;
- degraded room-state and audio modes that keep the room usable.

The local acceptance command is `pnpm run test:m0.5`.

## Manual Acceptance

Manual acceptance is reserved for behavior that cannot be verified reliably by automation:

- real VR headset movement and head turning;
- subjective voice audibility;
- subjective spatial-audio direction and distance;
- a 10 minute shared-room session with desktop, another browser or mobile device, and VR.

Manual acceptance must run only after the automatic suite is green.

## Closure Criteria

M0.5 is complete only when:

- `pnpm run lint` passes;
- `pnpm run typecheck` passes;
- `pnpm run build` passes;
- `pnpm run test` passes;
- `pnpm run test:m0.5` passes;
- final manual VR and audio acceptance passes;
- staging is deployed and verified for the accepted commit.
